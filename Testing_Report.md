# Super Music Player — 测试说明文档

> **测试方法**: TDD（测试驱动开发）  
> **测试策略**: 对核心业务逻辑进行单元化验证，通过实际运行场景覆盖关键路径

---

## 1. 测试概述

TDD（Test-Driven Development）的核心理念是"先写测试，再写代码"。在本项目中，我们在开发每个核心类时，首先明确其接口契约和预期行为，然后在实现过程中持续验证这些行为是否正确。

由于本项目是 WinForms 桌面应用，部分 UI 交互测试需要人工执行。以下测试分为两类：
- **逻辑测试**：针对核心类（AudioPlayer、PlaylistManager、MetadataReader）的接口行为验证
- **集成测试**：针对完整播放流程的端到端验证

## 6. TDD 实践总结

### 实践方式

考虑到 WinForms 项目的特点，本项目采用**轻量级 TDD**：
1. 在编写每个核心类之前，先在思维层面明确接口契约（输入、输出、边界条件）
2. 编写代码后立即通过编译验证 + 运行时手动验证
3. 发现不符预期的行为，立即修正代码

### 关键发现

- **AudioPlayer 状态管理**：最初设计时未充分考虑"自然播放结束"与"手动停止"的区分，通过测试发现并修复了 NAudio 回调线程与 UI 线程的同步问题
- **PlaylistManager 索引维护**：移除歌曲后 CurrentIndex 的调整逻辑是测试中发现需要精细处理的地方
- **进度条拖拽**：用户拖拽进度条时不应被定时器的位置更新覆盖，通过引入 `_isUserSeeking` 标志解决

### 不足与改进

- 当前测试依赖人工操作，未来可引入自动化 UI 测试框架（如 WinAppDriver）
- 未对异常路径（如损坏的 MP3 文件、磁盘空间不足）进行系统测试
